# Big Data EMR / Delta Extension For CCE

This note explains how the existing CCE feature-platform PoC can be extended into a more explicit big-data processing design using Spark, Delta Lake, Airflow/MWAA, S3, Glue/Athena and EMR Serverless or EMR on EKS.

## Existing CCE vs Big-Data Extension

| Area | Existing `cce-feature-platform` | Big-data extension |
| --- | --- | --- |
| Main story | Customer Campaign Engine / CDP feature platform | Large-scale feature computation and data-lake operations |
| Local runtime | Python, SQLite, CSV/JSONL, FastAPI | PySpark, Delta Lake, S3-style layout, Airflow DAGs |
| Data size | Small deterministic sample for runnable demos | Synthetic scale data from 10K users to 480K+ users and 100GB+ benchmark datasets |
| Batch pipeline | Local Bronze -> Silver -> Gold plus Databricks mapping | Explicit Spark jobs for Bronze ingest, Silver feature engineering, Gold segmentation and anomaly detection |
| Online path | JSON-backed Redis stand-in, CDC simulation, Feature API | Same online design, backed by Redis/ElastiCache and CDC/stream updates |
| Deployment | Databricks job template, EKS manifests, Terraform templates | EMR Serverless for development, EMR on EKS for production-style Spark workloads |
| Governance | Data quality issues, lineage endpoint, drift output | Adds data volume, freshness, partitioning, file layout, Glue catalog and Airflow SLA concerns |

The local CCE implementation and the big-data extension have separate responsibilities. The local project keeps the domain model, identity resolution, feature serving and API flow easy to run. The big-data extension maps the same feature logic to distributed Spark jobs for larger datasets, backfills and scheduled batch windows.

## Synthetic Data Strategy

Use two kinds of synthetic data because they answer different questions.

| Data strategy | Best for | Notes |
| --- | --- | --- |
| Domain-specific Spark + Faker data | CCE feature, identity and campaign story | Generate customers, identities, transactions, policies, campaign events and CDC updates with realistic skew and timestamps. |
| TPC-DS / TPC-H benchmark data | Engine and storage benchmark | Useful for Spark, EMR, Athena, Trino or ClickHouse/Doris performance tests, but it is less tied to CCE business semantics. |
| Small deterministic fixtures | Local regression tests | Keep current sample data so the repo remains easy to run on a laptop. |

Recommended scale tiers:

| Tier | Users | Events / transactions | Purpose |
| --- | ---: | ---: | --- |
| Local | 10-100 users | hundreds | Fast tests and API demo |
| Functional scale | 10K users | 100K transactions | Validate Spark job logic and partition layout |
| Rollout scale | 480K active users | 1M-5M daily events | Validate feature-store sizing, Redis memory, CDC lag and batch SLA |
| Benchmark scale | 100GB+ | generated by TPC-DS/TPC-H or domain generator | Validate Spark/EMR storage and query performance |

## Suggested Script Layout

```text
cce-feature-platform/
  deploy/
    emr_delta/
      0_generate_synthetic_data.py
      1_bronze_ingest.py
      2_silver_feature_eng.py
      3_gold_segmentation.py
      4_anomaly_detection.py
      config.py
      requirements.txt
    airflow/
      cce_feature_pipeline_dag.py
```

| Script | Responsibility | Output |
| --- | --- | --- |
| `0_generate_synthetic_data.py` | Generate CCE domain data with Spark + Faker | Raw JSON/CSV/Parquet under `s3://.../raw/cce/` |
| `1_bronze_ingest.py` | Read raw data, add metadata, normalize timestamps and source file names | Delta Bronze tables |
| `2_silver_feature_eng.py` | Resolve identity, clean invalid rows, build transaction and policy facts | Delta Silver tables and DQ reject tables |
| `3_gold_segmentation.py` | Build customer features, policy features, K-Means segments and campaign eligibility | Delta Gold feature and segment tables |
| `4_anomaly_detection.py` | Detect transaction outliers, customer feature anomalies and feature drift | Delta Gold monitoring and anomaly tables |
| `cce_feature_pipeline_dag.py` | Orchestrate the jobs with retries, SLA and dependency order | Airflow / MWAA pipeline |

This should not replace `src/cce_platform/pipeline.py`. The local pipeline remains the runnable reference; the Spark scripts show the production-scale execution model.

## Spark / Delta Design Points

| Design point | Recommendation |
| --- | --- |
| Storage | Use S3 as the durable data lake and Delta Lake for ACID table state. |
| Catalog | Register Bronze, Silver and Gold tables in Glue Catalog or Unity Catalog depending on runtime. |
| Partitioning | Partition large facts by `business_date` or ingestion date. Avoid high-cardinality partitions such as customer ID. |
| Clustering / Z-order | Cluster or Z-order Gold tables by `unified_customer_key`, `policy_id` and `business_date` for feature lookup and backfills. |
| File size | Compact small files after streaming or frequent incremental writes. Aim for query-friendly Parquet file sizes rather than thousands of tiny files. |
| Time columns | Keep both `event_time` and `ingest_time`. Event time drives business windows; ingest time supports freshness and late-data monitoring. |
| Idempotency | Use `event_id` when available, otherwise `business_key + event_time + source_sequence` or a `record_hash`. |
| Reprocessing | Design jobs so a partition or business date can be rebuilt deterministically. |

## Runtime Choice

| Runtime | Best fit | Why |
| --- | --- | --- |
| Databricks | Managed lakehouse, Unity Catalog, MLflow, Delta workflows | Already mapped in `deploy/databricks/cce_medallion_job.py`; best for governed feature engineering and MLOps. |
| EMR Serverless | Low-ops development and short-lived Spark jobs | Good for validating Spark scripts without managing clusters. |
| EMR on EKS | Production-style Spark jobs in a Kubernetes estate | Reuses EKS operations, supports multiple Spark versions and can mix on-demand and Spot capacity. |
| EMR on EC2 | Traditional managed Hadoop/Spark cluster | Useful when the team wants more direct cluster control. |

The operating model is: local Python for fast validation, Databricks for governed lakehouse and MLflow workflows, and EMR on EKS or EMR Serverless for AWS-native Spark operations.

## Capacity And Cost Model

The existing CCE sizing already covers Redis, MSK, Debezium, stream jobs and Feature API HPA. The big-data extension adds the batch-compute side:

| Component | Sizing logic |
| --- | --- |
| S3 / Delta storage | Data volume grows with raw events, Silver facts, Gold features, history retention and table versions. |
| Spark compute | Driven by input size, shuffle volume, feature windows, join skew and partition count. |
| Airflow / MWAA | Driven by DAG count, schedule frequency, retries, SLA and alerting requirements. |
| Glue / catalog | Driven by table count, partition count, metadata operations and access-control needs. |
| Athena / BI query | Driven by scanned bytes, file format, partition pruning and table compaction. |

For 480K active users, the online feature-store memory may still be modest, but the offline platform must handle historical events, replay, feature windows, model training sets and backfills. That is where Spark/Delta/Airflow becomes more important than the API code.

## Repository Extension Artifacts

The base CCE project focuses on domain logic, identity resolution, online feature serving and deployment templates. The optional artifacts below keep the distributed Spark runtime separate from the local demo while documenting how the same pipeline runs at larger scale:

1. `deploy/emr_delta/0_generate_synthetic_data.py`: Spark synthetic data generator for customers, identities, transactions, policies and campaign events.
2. `deploy/emr_delta/1_bronze_ingest.py` through `4_anomaly_detection.py`: EMR/Delta jobs that mirror the local Bronze -> Silver -> Gold flow.
3. `deploy/airflow/cce_feature_pipeline_dag.py`: Airflow/MWAA DAG skeleton with dependencies, retries, execution timeout and EMR Serverless job submission.
4. `deploy/emr_delta/README.md`: S3/Delta layout notes covering raw, Bronze, Silver, Gold, partitioning, replay and table registration.
5. Capacity notes in this document and `REALTIME_FEATURE_PLATFORM_480K.md`: 480K-user feature size, Redis memory, daily event volume, Spark input size and batch-window considerations.
6. `../data-governance-poc`: executable checks for schema, freshness, volume, timestamp deviation, duplicate events and reconciliation.

The boundary is intentional: `src/cce_platform/` remains the local runnable reference, while `deploy/emr_delta/` and `deploy/airflow/` describe the distributed runtime shape. This keeps laptop-friendly code separate from EMR/Airflow deployment artifacts.

For rollout constraints, cost drivers and maturity gaps, see `OPERATIONS_MATURITY_AND_COST.md`.
